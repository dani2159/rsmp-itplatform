Prinsip Form SIMRS

Form di sistem informasi rumah sakit harus mendukung workflow nyata petugas, bukan sekadar mengikuti struktur database. Semua field wajib punya tujuan klinis/operasional yang jelas.

Pengalaman Form Tanpa Hambatan
01 · Efisien

Data penting terlihat dulu

Prioritaskan NIK, No. RM, nama pasien, penjamin, dokter, tanggal layanan, diagnosis, dan status kunjungan.

02 · Aman

Validasi sebelum simpan

Gunakan validasi realtime untuk format, tapi jangan menghambat input cepat kecuali risiko tinggi.

03 · Jelas

Label bukan placeholder

Label selalu tampil. Placeholder hanya contoh format, bukan pengganti instruksi.

04 · Traceable

Field penting siap audit

Perubahan data medis, obat, billing, dan klaim perlu tercatat dengan pengguna, waktu, dan alasan perubahan.

Struktur field

Rule penempatan informasi

LabelSelalu tampil di atas field. Gunakan 14px, font weight 500, dan kalimat pendek: Nama Pasien, No. RM, Tanggal Lahir.
WajibGunakan tanda * untuk data wajib. Jangan menandai semua field sebagai wajib.
PlaceholderBerisi contoh format: 3273xxxxxxxxxxxx, HH:mm, 0301R0010626V000001.
HelperBerisi batasan, contoh, atau akibat input. Muncul sebelum error.
ErrorSpesifik dan dapat diperbaiki. Hindari “Data tidak valid” tanpa penjelasan.
Aksi metaTaruh aksi kecil di kanan label: Cek BPJS, Scan Barcode, Generate No. RM.

Validasi & Pesan Error

Validasi harus membantu pengguna memperbaiki data. Jangan membuat form terasa seperti menolak pekerjaan pengguna.

Validation timing

Kapan validasi muncul

  • Saat input: format sederhana seperti NIK 16 digit, angka saja, email, dan nomor HP.
  • On blur: data yang perlu request API seperti duplicate NIK, cek BPJS, cek stok.
  • On submit: validasi lintas-field seperti tanggal masuk ≤ tanggal keluar.
  • Blocking: hanya untuk risiko klinis, billing, klaim, atau data wajib.

Layout Form

Layout form harus responsif, cepat dibaca, dan cocok untuk desktop rumah sakit ≥1366px, tablet ≥768px, serta mobile read-only pimpinan.

Form utama desktop

2–4 kolom

Gunakan 4 kolom untuk field pendek seperti NIK, BPJS, tanggal, penjamin. Gunakan 2 kolom untuk field sedang seperti nama, alamat, dokter.

Kelompokkan: Identitas → Tanda vital → Anamnesis → Diagnosis → Order → Rencana.

Aksesibilitas & Keyboard

Semua input harus bisa dipakai dengan keyboard, screen reader, dan tetap jelas tanpa mengandalkan warna saja.

Yang disarankan

Gunakan pattern ini

  • Label selalu terhubung ke input melalui for dan id.
  • Gunakan aria-describedby untuk helper/error text.
  • Indikator focus jelas minimal 2px + offset.
  • Aksi icon-only wajib punya aria-label.
  • Teks error harus menjelaskan cara memperbaiki input.
Yang dihindari

Hindari pattern ini

  • Placeholder sebagai satu-satunya label.
  • Membedakan status hanya dengan warna merah/kuning/hijau tanpa teks.
  • Nonaktif untuk field yang perlu bisa dicopy; gunakan readonly.
  • Dropdown panjang tanpa search untuk dokter, ICD, obat, atau pasien.
  • Validasi API setiap ketikan tanpa debounce.

Implementasi CSS, Tailwind, dan React

Input memakai struktur .field, .label-line, .input, .input-wrap, .with-left, .with-right, dan .helper. Dropdown memakai .ds-dropdown.

function TextField({ id, label, required, leftIcon, error, className = '', ...props }) {
  return (
    <div className="field">
      <div className="label-line">
        <label htmlFor={id}>{label}{required && <span className="required">*</span>}</label>
      </div>
      <div className="input-wrap">
        {leftIcon && <span className="field-icon">{leftIcon}</span>}
        <input id={id} className={`input is-md ${leftIcon ? 'with-left' : ''} ${className}`} {...props} />
      </div>
      {error && <span className="helper error">{error}</span>}
    </div>
  )
}
Catatan: Jangan membuat ulang input dengan utility inline di HTML. Pertahankan selector komponen supaya style konsisten di semua halaman.
